# Отчет по лабораторной работе: Реализация SOA-калькулятора

## 1. Цель работы
Разработать десктопное приложение (Win-интерфейс) в виде калькулятора и вынести всю бизнес-логику вычислений во внешний Web-сервис. Реализовать взаимодействие клиента и сервера по протоколу HTTP. Продемонстрировать работу архитектуры SOA (Service-Oriented Architecture) с использованием виртуализации (Vagrant) для изоляции серверной части.

## 2. Архитектура решения (SOA)

В рамках работы реализована следующая архитектура:

| Компонент | Технология | Роль в SOA |
| :--- | :--- | :--- |
| **Клиент (UI)** | Python + PyQt5 | Отвечает только за отображение интерфейса, сбор ввода пользователя и отправку HTTP-запросов. Не содержит логики вычислений. |
| **Транспорт** | HTTP / JSON | Протокол обмена данными между клиентом и сервисом. |
| **Сервер (Business Logic)** | Python + FastAPI | Принимает выражения, выполняет безопасные вычисления, возвращает результат. |
| **Среда исполнения сервера** | Ubuntu 20.04 (Focal) в Vagrant VM | Изолированная среда, эмулирующая удаленный сервер. |

**Преимущества выбранной архитектуры:**
*   **Разделение ответственности:** Клиент можно переписать на любой другой фреймворк (WPF, Electron, Android), не меняя сервер.
*   **Безопасность:** Логика вычислений находится на сервере, клиент не имеет доступа к системным ресурсам сервера.
*   **Масштабируемость:** Серверный слой можно горизонтально масштабировать (запустить несколько копий сервиса).

## 3. Реализация Web-сервиса (Серверная часть)

Сервер реализован на базе фреймворка **FastAPI**.

### 3.1. Бизнес-логика
Вместо небезопасного `eval()`, на сервере используется парсер выражений на базе модуля `ast` (Abstract Syntax Trees). Это позволяет выполнять математические операции (`+`, `-`, `*`, `/`, скобки), но запрещает выполнение произвольного кода.

### 3.2. Настройка согласно заданию (Пункт 3)
В коде сервиса (`main.py`) реализованы требования к настройкам метода:
1.  **Буферизация ответов:** Обеспечивается самим фреймворком FastAPI/Uvicorn (ответ формируется полностью перед отправкой).
2.  **Отключение кэширования:** В обработчике запроса явно устанавливаются заголовки:
    ```python
    response.headers["Cache-Control"] = "no-cache, no-store, must-revalidate"
    response.headers["Pragma"] = "no-cache"
    ```
3.  **Отключение состояния сеансов (Stateless):** Сервис не использует глобальные переменные для хранения состояния пользователей. Каждый запрос обрабатывается независимо.

**API эндпоинт:**
*   Метод: `POST`
*   URL: `/calculate`
*   Входные данные (JSON): `{"expression": "1+2*3"}`
*   Выходные данные (JSON): `{"result": 7, "error": null}`

## 4. Реализация Клиентского приложения (GUI)

Интерфейс разработан с использованием библиотеки **PyQt5**.
*   **Функционал:** Стандартный калькулятор с кнопками цифр, операторов и очистки.
*   **Интеграция:** При нажатии кнопки `=` выражение отправляется на сервер через библиотеку `requests`.
*   **Обработка ошибок:** Если сервер недоступен (например, выключена VM), приложение корректно отображает ошибку, не падая.

## 5. Развертывание на виртуальной машине (Vagrant)

Для выполнения требования об изоляции среды и эмуляции удаленного сервера использован инструмент **Vagrant** с гипервизором **VirtualBox**.

### 5.1. Конфигурация окружения
*   **ОС:** Ubuntu 20.04 LTS (`ubuntu/focal64`).
*   **Сеть:** Настроен проброс портов (`Forwarded Port`).
    *   Порт внутри VM: `8000` (слушает FastAPI).
    *   Порт на хосте (ПК студента): `8001`.
    *   *Это позволяет запускать сервис внутри VM, а обращаться к нему с основной ОС как к удаленному ресурсу.*
*   **Автоматизация:** Скрипт `provision.sh` автоматически устанавливает Python 3, pip, зависимости и запускает сервис в фоновом режиме.

### 5.2. Структура проекта
```text
lab_1/
├── calculator_api/          # Исходный код Web-сервиса
│   ├── main.py              # Логика сервиса
│   └── requirements.txt     # Зависимости (fastapi, uvicorn)

│   └── calculator_gui.py    # Код приложения PyQt5

```

# Контрольные вопросы и ответы

1. Чем отличаются Web-сервисы от Web-сайтов?

    Web-сайт ориентирован на человека (Human-oriented). Его главная цель — визуальное представление информации (HTML-страницы). Пользователь взаимодействует с сайтом через браузер, кликая по ссылкам и заполняя формы.
    Web-сервис ориентирован на машину (Machine-to-machine). Его цель — предоставление функционала или данных другой программе. Он не генерирует красивый HTML, а возвращает структурированные данные (обычно JSON или XML) в ответ на программный запрос.
        Пример из проекта: Наш calculator_api — это Web-сервис: он не отдает HTML-страницу, а принимает JSON {"expression": "2+2"} и возвращает JSON {"result": 4}. Калькулятор на PyQt5 выступает в роли программы-клиента, которая «понимает» этот JSON.


2. Каков язык передачи данных Web-сервисов и Web-сайтов?
    Для Web-сайтов: Основным языком передачи данных является HTML (HyperText Markup Language), который браузер интерпретирует для отрисовки интерфейса. Часто дополняется CSS (стили) и JavaScript (логика интерфейса).
    Для Web-сервисов: Основными форматами данных являются JSON (JavaScript Object Notation) и XML (eXtensible Markup Language). Они удобны для машинной обработки.
        Пример из проекта: В нашем решении используется JSON. FastAPI автоматически сериализует объекты Pydantic в JSON, а библиотека requests в GUI легко парсит этот JSON обратно в переменные Python.


3. Назначение протоколов SOAP и WSDL?

    SOAP (Simple Object Access Protocol): Это протокол обмена сообщениями, строго регламентирующий структуру XML-сообщения (заголовок, тело, ошибки). Он часто используется в крупных корпоративных системах, где важна строгая типизация и безопасность.
    WSDL (Web Services Description Language): Это XML-файл, который описывает интерфейс Web-сервиса, работающего по протоколу SOAP. Он говорит клиенту: «Вот какие методы есть у сервиса, какие параметры они принимают и какие данные возвращают». По сути, это контракт между клиентом и сервером.
        Нюанс проекта: В данной лабораторной работе мы использовали архитектуру REST (а не SOAP). REST не требует WSDL; описание API происходит через документацию (Swagger/OpenAPI, которая генерируется FastAPI автоматически по адресу /docs). Это более легковесный подход для современных микросервисов.



4. Каковы задачи посредников (прокси) в приложениях для общения с Web-сервисами?

    Посредники (Proxy, API Gateway, ESB) решают следующие задачи:
    Маршрутизация: Перенаправление запросов к нужному экземпляру сервиса (важно при масштабировании).
    Балансировка нагрузки: Распределение запросов между несколькими серверами для повышения производительности.
    Безопасность и авторизация: Проверка токенов доступа до того, как запрос попадет на бизнес-логику.
    Кэширование: Хранение частых ответов для ускорения работы.
    Трансформация протоколов: Например, конвертация SOAP-запроса в REST-вызов.
        Связь с проектом: В текущем учебном примере прокси не используется, так как система проста. Однако в реальной SOA архитектуре (например, на базе Nginx или Kong) такой прокси стоял бы перед нашим FastAPI сервисом, принимая все запросы на порту 80 и распределяя их.


5. Что хорошего и плохого при включении сеанса (Session) Web-сервиса?

    Плюсы:

        Сохранение контекста: Можно хранить состояние пользователя между запросами (например, содержимое корзины покупок или шаги мастера настройки).
        Персонализация: Легче подгружать данные, специфичные для текущего пользователя.

    Минусы:

        Нарушение принципа Stateless: Один из главных принципов REST и SOA — каждый запрос должен содержать всю необходимую информацию. Сессии делают сервис зависимым от предыдущего состояния.
        Проблемы масштабирования: Если сервис работает на нескольких серверах, сессия, созданная на сервере А, будет недоступна на сервере Б (требуется внешнее хранилище сессий, например Redis).
        Утечки ресурсов: Незакрытые сессии занимают память сервера.
        Пример из проекта: Мы отключили сессии (реализовали Stateless подход). Наш калькулятор не запоминает историю вычислений на сервере. Это упрощает развертывание в Vagrant и позволяет легко запустить 10 копий сервиса без сложной синхронизации сессий.



6. В чём суть SOA?

    SOA (Service-Oriented Architecture) — это парадигма проектирования программного обеспечения, при которой приложение строится не как единый монолит, а как набор слабосвязанных, независимых сервисов.
    Ключевые черты:
    Слабая связанность (Loose Coupling): Изменение в одном сервисе не ломает другие.
    Переиспользуемость: Один сервис (например, «Калькулятор») может использоваться разными клиентами (веб-сайт, мобильное приложение, десктоп).
    Стандартизация: Взаимодействие происходит через четко определенные интерфейсы (API).
        Реализация в проекте: Бизнес-логика (FastAPI) полностью отделена от интерфейса (PyQt5). Мы можем заменить PyQt5 на мобильное приложение на Flutter, и сервер даже не узнает об этом изменении.


7. Каковы цели, преследуемые при построении SOA?

   Основные цели внедрения SOA:
    Гибкость бизнеса: Быстрое реагирование на изменения требований рынка за счет замены отдельных сервисов.
    Повторное использование компонентов: Снижение затрат на разработку за счет использования готовых сервисов.
    Интеграция разнородных систем: Возможность объединить старые legacy-системы с новыми приложениями через единый стандарт взаимодействия.
    Независимое развитие: Разные команды могут разрабатывать и обновлять разные сервисы параллельно.
        В контексте лабы: Цель достигнута тем, что мы можем обновлять алгоритм вычислений в main.py (добавлять новые функции), не трогая код графического интерфейса.


8. Каковы уровни абстракции SOA в информационной системе предприятия?

    Обычно выделяют следующие уровни (слои):
    Уровень представления (Presentation Layer): Интерфейс для пользователя (веб, мобильное, десктоп приложение). В проекте: PyQt5 калькулятор.
    Сервисный уровень (Service Layer): Набор сервисов, предоставляющих бизнес-функционал. Это «кирпичики» системы. В проекте: FastAPI сервис /calculate.
    Уровень бизнес-процессов (Business Process Layer): Оркестрация сервисов. Здесь описывается, как несколько сервисов работают вместе для выполнения сложной задачи (например: «Оформить заказ» = «Проверить наличие» + «Рассчитать стоимость» + «Создать накладную»).
    Уровень интеграции (Integration Layer): Шины данных (ESB), брокеры сообщений (Kafka, RabbitMQ), обеспечивающие надежную доставку сообщений между сервисами.
    Уровень данных (Data Layer): Базы данных, файловые хранилища, внешние API.
        Применимость: В нашей лабораторной работе реализованы первые два уровня. Мы создали сервисный уровень (API) и клиентский уровень (GUI), демонстрируя базовую связку SOA.
